# Retrieval-Augmented Generation (RAG) 技术指南

## 什么是 RAG

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将信息检索与大语言模型生成能力相结合的 AI 架构。与纯粹依赖模型训练数据的方式不同，RAG 在每次回答问题前，先从外部知识库中检索相关内容，再将这些内容作为上下文传给模型，从而生成更准确、更可靠的回答。

RAG 的核心优势在于：模型不需要记住所有知识，只需要具备"理解和综合"的能力，而知识可以存储在外部数据库中并随时更新。

## RAG 的整体架构

RAG 系统由两个主要阶段组成：**索引阶段（Indexing）**和**检索生成阶段（Retrieval & Generation）**。

索引阶段是离线处理过程，将原始文档转化为可检索的向量。检索生成阶段是在线处理过程，响应用户的实时查询。

## 索引阶段

### 文档加载（Document Loading）

文档加载是 RAG 流程的起点。系统需要支持多种文件格式：

- **Markdown**：技术文档、README、Wiki 页面等，天然具有层级标题结构
- **HTML**：网页内容，需要解析 DOM 树提取有效文本，忽略导航栏、脚本等噪声
- **PDF**：研究论文、报告、合同等，文本提取后通常为平文本，缺少结构化标题
- **图片（Image）**：截图、扫描件等，需要 OCR 或 Vision LLM 先转化为文本

不同格式的文档有不同的特点，加载器（Loader）需要做格式归一化处理。

### 文本分块（Chunking）

由于 LLM 的上下文窗口有限，且过长的文本会稀释关键信息，必须将文档切分成适当大小的块（Chunk）。分块策略的选择直接影响检索质量：

**按标题分块（Heading-based Chunking）**适用于结构化文档（Markdown、HTML）。每个 Markdown 标题（`#`、`##`、`###`）标志着一个主题的开始，按标题切分能保留完整的语义单元。这种方法产生的块大小不均匀，但语义完整性高。

**固定大小分块（Fixed-size Chunking）**适用于平文本（PDF 提取文本、纯文本文件）。将文本按固定字符数切分，配合**滑动窗口重叠（Overlap）**避免跨块的关键信息丢失。例如：块大小 500 个字符，重叠 50 个字符，则相邻块共享 50 个字符，确保边界处的句子被完整包含在至少一个块中。

### 向量化（Embedding）

将每个文本块转化为高维浮点向量，这个过程称为 Embedding。这是 RAG 的核心技术之一。

**Embedding 模型**将文本的语义信息压缩到一个固定维度的向量空间中。语义相近的文本在向量空间中距离较近，这使得"相似性搜索"成为可能。

常用的 Embedding 模型：

- **OpenAI text-embedding-3-small**：1536 维，性价比高，适合生产环境
- **OpenAI text-embedding-3-large**：3072 维，精度更高，成本更高
- **Ollama nomic-embed-text**：768 维，本地运行，无需 API Key，适合离线场景
- **Ollama mxbai-embed-large**：1024 维，本地运行，效果更好

### 向量存储（Vector Storage）

生成的向量连同原始文本和元数据一起存储在向量数据库中。向量数据库专门针对高维向量的近似最近邻（ANN）搜索进行了优化。

**Qdrant** 是一个高性能的开源向量数据库，支持：
- 余弦相似度（Cosine Similarity）搜索
- 负载过滤（Payload Filtering）——按元数据过滤搜索结果
- gRPC 和 REST 两种 API

## 检索生成阶段

### 查询向量化

用户输入的查询文本，使用**与索引阶段相同的 Embedding 模型**转化为查询向量。这是关键约束：查询向量必须与存储的文档向量在同一个向量空间中，才能进行有意义的相似度比较。

### 相似度检索

将查询向量发送给向量数据库，检索最相似的 Top-K 个文本块。相似度通常用**余弦相似度**衡量，值域 [0, 1]，分数越高代表语义越接近。

检索的 K 值（topK）是一个重要超参数：
- 太小（如 1）：可能错过关键信息
- 太大（如 20）：引入噪声，且超出 LLM 上下文窗口

通常 3-5 是一个合理的起点。

### 上下文组装（Context Assembly）

将检索到的 K 个文本块拼接成一个结构化的上下文，作为 LLM 的 System Prompt 的一部分。这个步骤也称为 Prompt Engineering。

典型的 RAG Prompt 结构：
```
System: 你是一个助手，请仅基于以下上下文回答问题。

=== 上下文 ===
--- Chunk 1 ---
[检索到的第一段文本]

--- Chunk 2 ---
[检索到的第二段文本]
=== 结束 ===

User: [用户的原始问题]
```

### 答案生成（Generation）

将组装好的 Prompt 发送给 LLM（如 GPT-4o-mini 或本地 Llama3），模型基于上下文生成回答。由于上下文来自最新的文档知识库，模型能够回答训练数据中不存在的问题。

## RAG 的常见优化方向

### Chunk 策略优化

- **父子分块（Parent-Child Chunking）**：检索时用小块（精准），生成时用大块（完整上下文）
- **句子级分块**：按句子边界切分，语义更完整
- **语义分块（Semantic Chunking）**：利用 Embedding 相似度动态决定分块边界

### 检索优化

- **混合搜索（Hybrid Search）**：向量搜索 + 关键词搜索（BM25）结合，提升召回率
- **重排序（Reranking）**：用专门的 Reranker 模型对初步检索结果重新打分排序
- **查询扩展（Query Expansion）**：用 LLM 将用户问题改写为多个查询，提高召回率
- **HyDE（Hypothetical Document Embedding）**：先让 LLM 生成假设答案，再用假设答案检索

### 生成优化

- **引用溯源**：在回答中标注每个信息点来自哪个 Chunk
- **置信度控制**：当检索分数低于阈值时，告知用户"知识库中没有相关信息"
- **对话历史**：多轮对话中保留历史消息，让 LLM 理解上下文

## 向量与相似度的数学基础

向量（Vector）是由一组浮点数组成的数组，可以理解为高维空间中的一个点。文本 Embedding 将語义信息映射到这个高维空间，使得"意思相近"的文本在空间中聚集。

**余弦相似度**计算两个向量夹角的余弦值：

- 夹角为 0（方向相同）→ 余弦值 = 1.0（完全相似）
- 夹角为 90°（正交）→ 余弦值 = 0.0（不相关）
- 夹角为 180°（方向相反）→ 余弦值 = -1.0（完全相反）

在语义搜索中，余弦相似度高于 0.8 通常表示强相关，0.5-0.8 表示中等相关。

## 本项目的实现

本项目用 Go 语言实现了一个完整的 RAG 学习框架，包含：

- **Loader 层**：支持 Markdown、HTML、PDF、图片（占位）
- **Chunker 层**：Heading 策略和 FixedSize 策略，均可通过 `--strategy` 参数切换
- **Embedder 层**：OpenAI 和 Ollama 双实现，通过接口抽象互相替换
- **VectorStore 层**：Qdrant 向量数据库，使用 gRPC SDK
- **Generator 层**：OpenAI GPT-4o-mini 和 Ollama 本地模型

运行索引：`go run ./cmd/index --file docs/example.md`
运行查询：`go run ./cmd/query --q "什么是 RAG？"`
